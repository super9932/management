파이프라인 컨텍스트

4-LAYER · BFF

본 명세는 4계층 응답 생성 파이프라인의 첫 BE 계층(nab BE) 을 기준으로 작성되었다. nab BE는 FE 와 직접 연결되는 BFF 이자 도메인 영속화의 단일 책임자이며, 그 외 계층은 nab BE 가 정의한 인터페이스를 따른다.

SYSTEM TOPOLOGY

nab BE 는 중앙 ink-filled 박스 — OAuth2 AS · Resource Server · NahClient · CoreService · Privacy/Audit · MyBatis 모듈 포함. nah / AI 는 사내 별도 프로세스 (파랑) — X-Request-Id 로만 추적되는 무상태 위임 계층. Gemini 는 외부 LLM (accent). 보험코어 는 외부 API only — 4종 Outbound API (보험증권 / 고객정보 / 상품정보 / FP정보) 를 read-through 호출하며 NAB 측 영속화는 없다. 영속화는 3개 인스턴스로 분리: On-Prem Counsel RDB (대화·메시지·Tool Call · counsel_*), NAB Cloud MySQL (인증·관리자·통계·감사 · oauth2_* / admin_* / usage_* / audit_log), Redis (RT · session).

진입 (OTT → AT) 흐름 — 상담AI FE 는 직접 OTT 를 발급받지 않는다. ① FP월드 FE 가 FP월드 BE 로 로그인, ② FP월드 BE 가 사전 합의 HMAC(auth.ott.api-keys)으로 NAB POST /v1/post/auth/ott/issue 호출하여 OTT 수령 (purple dashed), ③ FP월드 FE 가 OTT 를 상담AI FE 로 전달, ④ 상담AI FE 가 OTT 를 POST /v1/post/auth/oauth2/token (token-exchange RFC 8693) 에 제출해 AT/RT 발급 — 이후 모든 메인 API 는 Bearer AT 로 호출 (accent · HTTPS · SSE). 시퀀스 상세는 No.1 · No.3 참조.

  • 시간 포맷 (v0.23 변경): 모든 응답은 KST (UTC+09:00) 기준 고정.
    • 날짜 + 시간 → yyyy-MM-dd HH:mm:ss (예: 2026-05-19 19:45:11)
    • 날짜만 → yyyy-MM-dd (예: 2026-05-19)
    • 밀리초 · 타임존 suffix 미포함. ISO 8601 T 구분자 · UTC Z 사용 안 함.
    • 직렬화: Jackson JavaTimeModule + LocalDateTime/LocalDate + @JsonFormat(pattern=…, timezone="Asia/Seoul"). DB TIMESTAMP/DATETIME(6) 에서 읽은 값은 KST 로 변환 후 직렬화.
    • 요청(Request) 도 동일 포맷 권장. 보험코어 응답 의존 필드(saleStartDt 등 일부) 는 yyyy-MM-dd 로 정규화 — 보험코어 원본의 yyyyMMdd 는 NAB BE 직렬화 단계에서 변환.
  • 페이지네이션: page (1-based) + limit · 응답 meta.total
  • 대화 세션 정책 (v1.0): 마지막 사용자 질의 입력 시점을 기준으로 30분 idle = 세션 종료. 종료 트리거 5종 — (1) 30분 idle (2) [새로운 상담] 버튼 (3) 다른 상담 이력 선택 (4) FP월드 로그아웃 (5) FP월드 보안 세션 만료. 종료 시 마지막 사용자 질의가 포함된 대화는 1년간 DB 보관 (사용자 [삭제] 액션도 soft delete · counsel_conversation.is_active='N' 로 마킹) → 보관 기간 만료 시 별도 BO 배치 API 로 hard delete.
  • Soft delete 원칙: 사용자/관리자 삭제 시 is_active='N' 마킹만 수행 (감사 추적용). 조회 API 는 is_active='Y' 만 반환. 영구 삭제는 별도 admin batch endpoint 가 담당.
